* Overview

Phase-by-phase plan for the Agda string-diagram tool under
~src/Categories/APROP/Hypergraph/~.  Each phase builds on the
previous one.  The ultimate goal is to make every module typecheck
with ~--safe~ and have ~CategoricalCrypto.agda~ transitively import
them.

See https://arxiv.org/pdf/2604.17592 for high-level details.

* Active postulates

All three blocks live in files that currently omit ~--safe~ because
of these postulates.  The top-level ~CategoricalCrypto.agda~ does
*not* import any of them, so the library stays ~--safe~ in the
meantime.

** DONE [#C] ~soundness-axiom~ catch-all eliminated
   :PROPERTIES:
   :file:     src/Categories/APROP/Hypergraph/Soundness.agda
   :END:

   The omnibus ~soundness-axiom~ catch-all is gone.  Every one of
   the 19 atomic ≈Term axioms has its own explicit dispatch to a
   named focused lemma in ~SoundnessAxioms.agda~.  Of those 19:

   - *11 fully proved* (possibly modulo two internal subst₂-cancel
     postulates): idˡ, idʳ, id⊗id, λ⇐∘λ⇒, λ⇒∘λ⇐, ρ⇐∘ρ⇒, ρ⇒∘ρ⇐,
     α⇐∘α⇒, α⇒∘α⇐, σ∘σ, λ⇒∘id⊗f≈f∘λ⇒ (λ-nat).
   - *8 still postulated*: ρ⇒∘f⊗id≈f∘ρ⇒ (ρ-nat), α-comm, triangle,
     σ∘[f⊗g]≈[g⊗f]∘σ (σ-nat), hexagon, pentagon, assoc, ⊗-∘-dist.
     See "Refactor / 6." below for the per-axiom proof plan.

   Dispatch uses ~{-# INJECTIVE_FOR_INFERENCE ⟪_⟫ #-}~ (in
   ~Translation.agda~) plus ~--lossy-unification~ (on
   ~Soundness.agda~).  Without both flags, dispatching any focused
   postulate whose type mentions a `hComposeP ... (hTensor ...)`
   chain triggers a ~25-minute unification blow-up.  With both,
   every dispatch typechecks in <1s.

   Historical note (pre-~hComposeP~): the catch-all used to absorb
   18 constructors.  With the switch to pruned composition
   (~hComposeP~), the "LHS has more vertices than RHS" problem
   disappeared, and all axioms became reachable at the ≅ᴴ level.

   Example: ~idˡ : id ∘ f ≈Term f~.  At ~f : HomTerm A B~,
   - LHS ~⟪id ∘ f⟫ = hCompose ⟪f⟫ (hId B)~ has ~⟪f⟫.nV + length(flatten B)~ vertices.
   - RHS ~⟪f⟫~ has ~⟪f⟫.nV~ vertices.

   Since ~length(flatten B) > 0~ whenever ~B~ has any atom, the
   two cannot be in bijection; no refinement of the bijection
   functions can fix this.

   Discharging these axioms therefore requires one of the
   following architectural changes before the lemmas themselves
   can be written.

* Refactor

  Strategic refactoring plan to discharge the remaining 12 ≈Term
  axioms efficiently by maximizing reuse.  The key observation is
  that both `idˡ-proof` and `σ∘σ-proof` contain reasoning that only
  uses record fields of their `G`/`K` arguments, not any APROP-term
  structure.  Generalizing these to take arbitrary hypergraphs
  (rather than ~⟪f⟫~-specific ones) creates reusable building
  blocks for most remaining axioms.

  The sequence below is ordered by effort-to-unlock ratio: each
  item unlocks downstream axioms and is more tractable given the
  preceding items.

** DONE [#A] 1. Refactor ~idˡ-proof~ → ~hCompose-hId-R-iso~
   :PROPERTIES:
   :effort: 30 min
   :unlocks: generalized left-identity-composition
   :END:

   Current `idˡ-proof` module is parameterized by
   `(f : HomTerm A B)` but only uses `G = ⟪f⟫`'s record fields
   (nV, vlab, dom, cod, ein, eout, elab, …).  It does NOT depend
   on `f`'s syntactic structure.  Refactor the module header to:
   #+begin_src agda
     module hCompose-hId-R-proof
       {As : List X} {B : ObjTerm}
       (G : Hypergraph FlatGen As (flatten B))
       where
         ...
   #+end_src
   yielding a generic
   ~hCompose-hId-R-iso : hComposeP G (hId B) ≅ᴴ G~.

   `idˡ-sound` becomes a one-line specialization:
   `idˡ-sound f = hCompose-hId-R-iso ⟪f⟫`.

   The 230-line body of ~idˡ-proof~ transfers verbatim because
   all its helpers (cn≡0 via `hId-count-non-dom`, nE≡0 via
   `hId-nE`, `idˡ-cod-helper` via `hId-cod≡dom` +
   `hId-dom-Unique` + cast arithmetic, …) depend on `B`, not `f`.

   *Why this matters*: downstream axioms (λ-nat, triangle, some
   coherences) compose ⟪f⟫'s that are themselves complex
   expressions (e.g. `hTensor hEmpty ⟪f⟫` for `id⊗f`).  Having a
   generic `hCompose-hId-R-iso` lets us apply the identity law to
   those derived hypergraphs without re-deriving the proof.

** DONE [#A] 2. Prove ~hTensor-hEmpty-G-iso~
   :PROPERTIES:
   :effort: 2 h
   :unlocks: λ-naturality via chaining; partial progress on ρ-nat, triangle
   :END:

   Generalize `hTensor-hEmpty-hId-iso` from `hId A` to an
   arbitrary G:
   #+begin_src agda
     hTensor-hEmpty-G-iso
       : ∀ {As Bs} (G : Hypergraph FlatGen As Bs)
       → hTensor hEmpty G ≅ᴴ G
   #+end_src

   Type alignment holds: `(hTensor hEmpty G).As = [] ++ As = As`
   (definitional).

   *Fields*:
   - φ = φ⁻¹ = id.  Bijection laws = refl.
   - ψ = ψ⁻¹ = id.  Bijection laws = refl.
   - φ-lab = refl (splitAt 0 always gives inj₂, matches G.vlab).
   - ψ-ein / ψ-eout: routine `map-∘ / map-cong / map-id` (same
     as `hTensor-hEmpty-hId-iso`'s existing chain).
   - φ-dom / φ-cod: same routine chain.

   *Hard field: ψ-elab*.  Unlike the `hId A` specialization
   (where ψ-elab is vacuous because hId has no edges), a generic
   G may have edges.  The `(hTensor hEmpty G).elab e` value
   unfolds to `subst₂ FlatGen (map-via-raise vlab-injR …) … (G.elab e)`.
   Use the "strategic atom-ein" trick from `idˡ-proof`: choose
   `atom-ein = map-via-raise vlab-injR (G.ein e)` so the ψ-elab
   equation reduces to `refl` via the `with splitAt G.nE e` match.

   *Also provide mirror lemma* `hTensor-G-hEmpty-iso-substed`
   (see item 5).

** DONE [#A] 3. Prove ~hCompose-hId-L-iso~
   :PROPERTIES:
   :effort: 4 h
   :unlocks: idʳ directly; half of λ-nat, ρ-nat, α-nat via chaining
   :END:

   Port the `σ∘σ-proof` classify + mirror-witness structure to an
   arbitrary right-side hypergraph K:
   #+begin_src agda
     hCompose-hId-L-iso
       : ∀ {A : ObjTerm} {Bs} {K : Hypergraph FlatGen (flatten A) Bs}
       → Unique (Hypergraph.dom K)          -- side condition
       → hComposeP (hId A) K ≅ᴴ K
   #+end_src

   The side condition is supplied by `⟪_⟫-dom-unique` at the
   ~idʳ~ call site.

   *Structure* (template = `σ∘σ-proof` with G := hId A,
   K := arbitrary-with-Unique-dom):

   - C.nV = `(hId A).nV + count-non K.dom`.  φ splits via
     `splitAt (hId A).nV`:
     * inj₁ i ↦ `lookup K.dom (cast len-eq i)` (mirror into K).
     * inj₂ j ↦ `lookup (nonMem K.dom) j` (pruned-K-side).
     where `len-eq : (hId A).nV ≡ length K.dom` (via
     `hId-nV≡len-flatten` + `K.dom-ok` + `length-map`).

   - φ⁻¹ via `classify K.dom`:
     * inj₁ i ↦ `inject+ _ (cast (sym len-eq) i)`.
     * inj₂ j ↦ `raise (hId A).nV j`.

   - `φ-left`: case on splitAt; `classify-lookup-Unique` and
     `classify-lookup-nonMem` (newly added!) close the two
     branches.

      #+end_src
      and ~decode-attempt-Linear (f ⊗₁ g)~ now plumbs them through
      via ~decode-attempt-Linear f~ / ~decode-attempt-Linear g~.

      ~hTensor G K~ partitions vertices and edges into a disjoint
      union via ~injL~/~injR~.  Edges of ~G~ go first in the natural
      Fin order, then edges of ~K~.  Proof-shape sketch (the hard
      part):

      a. After processing ~G~'s edges, stack permutes to
         ~map injL G.cod ++ map injR K.dom~ (uses IH for ~G~).
      b. After processing ~K~'s edges, stack permutes to
         ~map injL G.cod ++ map injR K.cod = cod~ (uses IH for ~K~).
      c. ~extract-exact cod cod~ succeeds (foundation 8).

      Showing ~edge-step~ on a ~G~-edge with input
      ~map injL (G.ein eG)~ leaves the ~K~-side untouched *reuses
      ~hTensor-impl.ein-c-inj₁-red~* to collapse the combined ~ein-c~
      back to ~map injL (G.ein eG)~, then foundation (3, 6) (which
      themselves reuse ~↑ˡ≢↑ʳ~ / ~disj-L-R~).  The structural pattern
      is exactly the one from ~Congruence.hTensor-resp-≅ᴴ~'s
      case-analysis on ~splitAt G.nE e~.

      *The fundamental obstacle*: the IH ~∃ tG. decode-attempt G ≡ just tG~
      tells us decode-attempt G succeeds, but it doesn't *expose* the
      internal ~s_final~ that G's algorithm produced.  For the lifting
      argument we need either (i) a stronger IH that exposes G's
      ~s_final~ (e.g. ~∃ s_final t. process-all-edges G.dom ≡ (s_final, t) × s_final ↭ G.cod~),
      or (ii) a structural argument that bypasses G's IH and does
      direct algorithm reasoning on hTensor.  Option (i) requires
      changing the IH up the chain; option (ii) is essentially Step 4's
      compositional ~decode-hTensor~ lemma.

      ****** ~decode-attempt-hCompose G K~ (~1-2 weeks)

      ~hCompose G K~ identifies ~K.dom~ vertices with ~G.cod~ vertices
      via ~remap~ (so ~map remap K.dom ≡ map injL G.cod~).  Hardest case.

      Sub-steps:
      a. Process ~G~'s edges (analogous to ~hTensor~'s G-side, *reusing
         ~hCompose-impl.ein-c-inj₁-red~* etc. — these are the
         ~hCompose~ counterparts of the ~hTensor~ reductions).
      b. Crossover: after ~G~'s edges, stack ≅ ~map injL G.cod~.  By
         the ~map-remap-K-dom~ lemma (currently in ~Linearity.agda~'s
         ~hCompose-impl~ scope as an internal helper of
         ~Linear-hCompose~), ~map injL G.cod ≡ map remap K.dom~.  This
         lemma is already constructively proved — just needs to be
         exposed publicly.
      c. Process ~K~'s edges via ~remap~-translation: *reuse
         ~hCompose-impl.remap~ / ~remap-comm~* (already public).
      d. Final ~extract-exact (map remap K.cod) final-stack~ succeeds.

      Required new machinery beyond reuse:
      - A handful of foundation-style lemmas adapted to ~remap~
        instead of ~_↑ʳ_~ — analogous to (3, 6, 7) but with
        ~remap~ as the second injection.  Likely 1-2 days each.
      - Hoist ~map-remap-K-dom~ out of the ~Linear-hCompose~ where to
        the top level of ~Linearity.agda~ (or a new shared module).
      - A *topological soundness* argument: ~K~-edges processed after
        ~G~-edges always have their inputs in stack.  For ~⟪f⟫~ this
        is built into the smart constructors; for arbitrary linear ~H~
        it requires non-trivial justification.  The ~hCompose-impl~
        already has lemmas (~K-dom-Unique~, ~K-eb-noDom~) that capture
        the relevant facts; lift them to power the topological

**** Phase 4a.2 — Interface seeding
     :PROPERTIES:
     :effort: ~2 days, ~100 LOC
     :END:

     Given ~H, J : Hypergraph FlatGen As Bs~, seed a partial vertex
     bijection ~φ₀~ from the interfaces:
     #+begin_src agda
       seedFromInterfaces
         : (H J : Hypergraph FlatGen As Bs)
         → Maybe (PBij H.nV J.nV)
       seedFromInterfaces H J = do
         -- H.dom and J.dom both have length = length As, pointwise pair:
         φ₁ ← pairUp empty H.dom J.dom
         -- H.cod and J.cod both have length = length Bs:
         φ₂ ← pairUp φ₁ H.cod J.cod
         -- Optionally also check that H.vlab and J.vlab agree on paired

     #+end_src

     Returns ~nothing~ if:
     - A vertex in ~H.dom~ maps to two different ~J.dom~ entries
       (conflict).
     - Paired vertices have different ~vlab~ (label mismatch).

**** Phase 4a.3 — Edge matching + propagation
     :PROPERTIES:
     :effort: ~1 week, ~500 LOC
     :END:

     The core of the algorithm.  Given partial ~(φ, ψ)~, pick an
     unmatched edge ~e : Fin H.nE~ and try to match it to a ~J~ edge:
     #+begin_src agda
       matchEdge
         : (H J : Hypergraph FlatGen As Bs)
         → (pbij : PBij H.nV J.nV × PBij H.nE J.nE)
         → (e : Fin H.nE)
         → List (PBij H.nV J.nV × PBij H.nE J.nE)
                 -- all consistent extensions matching e to some e'∈J
     #+end_src

     *Algorithm*:
     1. Iterate over ~J.nE~.
     2. For each ~e' : Fin J.nE~, check:
        - Not already in ~ψ~'s range.
        - Label match: ~H.elab e ≡ J.elab e'~ (modulo atom-list
          equality, since labels are indexed by ~map vlab (ein)~
          atoms).  The decidable-equality machinery from 4a.0 does
          this at the ~mor~ level.
        - Arity match: ~length (H.ein e) ≡ length (J.ein e')~ and
          same for ~eout~.
     3. For each candidate ~e'~:
        - Try to extend ~φ~ pointwise by pairing ~H.ein e [i]~ ↔
          ~J.ein e' [i]~ for all ~i~, checking label consistency.
        - Similarly for ~eout~.
        - If all extensions succeed, add this ~(e, e')~ pair to ~ψ~
          and emit the extended ~(φ, ψ)~.
     4. Return the list of all successful extensions.

     *Propagation* is implicit here: matching one edge adds
     (~length (ein e) + length (eout e)~) new vertex constraints
     to ~φ~, which prune future edge-match choices.

**** Phase 4a.4 — Backtracking search
     :PROPERTIES:
     :effort: ~2 days, ~100 LOC
     :END:

     Glue ~matchEdge~ into a depth-first search:
     #+begin_src agda
       searchIso
         : (H J : Hypergraph FlatGen As Bs)
         → (fuel : ℕ)   -- bound to ensure termination
         → (pbij : PBij H.nV J.nV × PBij H.nE J.nE)
         → Maybe (CompleteBij H J)
       searchIso H J zero    pbij = nothing   -- fuel exhausted
       searchIso H J (suc k) pbij =
         if all-edges-matched pbij
           then just (extract-complete-bij pbij)
           else
             let nextEdge = pick-first-unmatched pbij in
             any-match (matchEdge H J pbij nextEdge)
                       (λ pbij' → searchIso H J k pbij')
     #+end_src

     Pick edges in natural order (~Fin.fromℕ~-style left-to-right).
     No heuristics — we care about simplicity, not speed.

     ~fuel~ is set to ~H.nE × J.nE~ (finite search tree upper
     bound).  Termination via ~fuel~ decrement; well-founded
     recursion on the search tree would be cleaner but more
     Agda-work.

**** Phase 4a.5 — Record assembly
     :PROPERTIES:
     :effort: ~1 week, ~400-600 LOC
     :END:

     Given a complete bijection ~(φ, φ⁻¹, ψ, ψ⁻¹)~ with the
     operational consistency invariants from ~PBij~, assemble the
     full 15-field ~_≅ᴴ_~ record:

     | Field                | Source                                       |
     |----------------------+----------------------------------------------|
     | φ, φ⁻¹               | PBij.forward / .backward                     |
     | φ-left, φ-rght       | PBij.consistent (expressed as equalities)    |
     | ψ, ψ⁻¹               | PBij (edges)                                 |
     | ψ-left, ψ-rght       | PBij.consistent for edges                    |
     | φ-lab                | maintained as an *invariant* of the search   |
     | ψ-ein, ψ-eout        | maintained — matched edges paired pointwise  |
     | φ-dom, φ-cod         | from 4a.2's interface seeding                |
     | atom-ein, atom-eout  | derived: `trans (cong (map K.vlab) ψ-ein) (map-∘ ...) (map-cong φ-lab) ...` |
     | ψ-elab               | hardest — see below                          |

     *The ψ-elab field*.  This is the edge-label compatibility
     claim:
     #+begin_src agda
       ψ-elab : ∀ e → subst₂ Gen (atom-ein e) (atom-eout e) (J.elab (ψ e))
                    ≡ H.elab e
     #+end_src

     From 4a.3's label match, we know ~H.elab e~ and ~J.elab (ψ e)~
     are propositionally equal at the generator level (after
     decidable equality succeeded).  But the iso needs to equate
     them *modulo* ~subst₂~ along the atom-list equalities.  The
     "strategic atom-ein" trick (used pervasively in
     SoundnessProved) chooses ~atom-ein~ and ~atom-eout~ to match
     the specific proof forms produced by ~matchEdge~'s label
     check, so ~ψ-elab~ reduces to ~refl~ or a single ~cong~.

     *Reuse opportunity*: the record-assembly boilerplate is
     similar to ~SigmaNat~'s final assembly (~1619-line file that
     builds a ~_≅ᴴ_~ record from a permutation of vertices).  We
     can factor out a ~mkIsoFromBij~ helper that packages the
     common pattern:
     #+begin_src agda
       mkIsoFromBij
         : (bij : CompleteBij H.nV J.nV)     -- φ with laws
         → (ebij : CompleteBij H.nE J.nE)    -- ψ with laws
         → (lab-ok : ∀ i → J.vlab (bij .fwd i) ≡ H.vlab i)
         → (ein-ok  : ∀ e → J.ein  (ebij .fwd e) ≡ map (bij .fwd) (H.ein e))
         → (eout-ok : ∀ e → J.eout (ebij .fwd e) ≡ map (bij .fwd) (H.eout e))
         → (dom-ok  : J.dom ≡ map (bij .fwd) H.dom)
         → (cod-ok  : J.cod ≡ map (bij .fwd) H.cod)
         → (elab-ok : ∀ e → ...ψ-elab shape...)
         → H ≅ᴴ J
     #+end_src
     Then ~findIso~ just threads the partial bijection's invariants
     into the shape required by ~mkIsoFromBij~.

     Critical path: 4a.0 → 4a.1 → 4a.2 → 4a.3 → 4a.4 → 4a.5 → 4a.6
     (mostly sequential, few parallel opportunities).

     *Safety*: everything in 4a can be ~--safe~.  No postulates
     or ~--lossy-unification~ required (the algorithm is purely
     computational; the record assembly is structural).

**** Risks

     - *Decidable equality for generators*.  ~APROPSignature~
       doesn't require it.  We avoid any invasive upstream change
       by putting the decidable-equality requirement on a new
       extension record ~APROPSignatureDec~ (4a.0) — all current
       modules keep working unchanged.  Only ~findIso~ and the
       ~smcat~ tactic are parameterised by ~APROPSignatureDec~;
       signatures without decidable equality lose only access to
       the tactic, not to any existing proof.

     - *ψ-elab proof construction*.  The atom-list equalities
       ~atom-ein~, ~atom-eout~ must be constructed in a form that
       makes ~ψ-elab~ reduce to something tractable.  If we try
       ~refl~-based strategic choices and they don't work (because
       matchEdge's label match uses different proof forms), the
       record-assembly step could blow up.  Fallback: use
       ~≅ᴴ~-reasoning or a more permissive equivalence internally.

     - *Fuel-based termination*.  Cleaner would be well-founded
       recursion on (unmatched edges, remaining search tree), but
       that's more Agda-work.  Fuel is simple and matches
       TensorRocq's Rocq implementation.

     *Runtime performance is intentionally deferred.*  The plan
     uses the simplest possible data structures (pure functions,
     lists) and no heuristics.  If ~findIso~ turns out to be too
     slow during type-checking for practical ~smcat~ usage, we'll
     profile and optimise then — potentially swapping ~PartialMap~
     for ~Vec (Maybe (Fin m)) n~, adding MRV-style heuristics, or
     emitting the iso via reflection code generation rather than
     in-type computation.

*** TODO Phase 4b — Reflection tactic ~smcat~
    :PROPERTIES:
    :effort: ~1-2 weeks
    :depends-on: Phase 4a + Phase 3.5 (completeness)
    :END:

    An Agda reflection macro that dispatches goals of the form
    ~f ≈Term g~:
    1. Reify the goal; extract HomTerms ~f~, ~g~.
    2. Compute ~⟪ f ⟫~ and ~⟪ g ⟫~ by ordinary Agda reduction.
    3. Apply ~findIso ⟪ f ⟫ ⟪ g ⟫~.
    4. On ~just iso~, emit the proof ~completeness iso~ (from
       Phase 3.5).  On ~nothing~, fail the tactic with a useful
       error.

    The tactic is *sound* (only emits proofs when the iso exists
    and completeness applies) and *partial-complete* (succeeds in
    practice for all cases arising in typical proofs).

    *Implementation notes*:
    - Agda's ~Reflection~ API gives access to ~Term~ / ~Type~
      manipulation; ~unquote~ splices back into the goal.
    - Performance: ~findIso~ runs entirely in Agda's evaluator
      during macro expansion; its runtime ≈ TensorRocq's reported
      "<1s on dozen-edge hypergraphs".
    - For debugging, expose a non-macro variant
      ~smcat-test : (f g : HomTerm A B) → Maybe (f ≈Term g)~.

*** Phase 4 dependency summary

    | Sub-phase | Depends on                  |
    |-----------+-----------------------------|
    | 4a        | Hypergraph.Core + Iso       |
    | 4b        | 4a + Phase 3.5 completeness |

    Critical path: Phase 3 → Phase 3.5 → 4a → 4b.

    *Why this is a win over the previous plan*: no tensor
    infrastructure needed for the ~smcat~ tactic.  Skipping
    tensors saves ~4 weeks of development.  Tensor semantics
    (proving `f ≈ g → ⟦f⟧ᵀ ≡ ⟦g⟧ᵀ` etc.) remains relevant for the
    rewrite tactic ~srw~ (Phase 5) — but not for the basic
    equality-by-connectivity tactic.

    *Alternative tactic (no Phase 3.5)*: implement term-level
    confluent rewriting on HomTerms using only the already-proved
    ≈Term axioms.  Each rewrite step emits an ~≈Term~ proof
    directly, so no appeal to completeness is needed.  Equivalent
    in power; harder to ensure confluence without a canonical
    form, but avoids the Phase 3.5 dependency entirely.
